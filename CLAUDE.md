# Working Rules (프로젝트 작업 규칙)

이 프로젝트에서 작업할 때는 아래 규칙을 반드시 따른다.

1. **배경/중요 대화 기록**: 프로젝트 배경과 중요한 논의 내용은
   `docs/ProjectContext.md`에 기록한다. 새로운 배경 정보나 방향 전환이
   생기면 해당 문서를 갱신한다.
2. **구현 전 Plan.md 작성**: 실제 코드 구현을 시작하기 전에 반드시
   `docs/Plan.md`를 작성(또는 갱신)하고, 그 계획에 따라 구현한다.
   계획 없이 바로 구현을 시작하지 않는다.
3. **기획 문서 독립 관리**: `docs/IA.md`(정보 구조),
   `docs/UserScenarios.md`(사용자 시나리오), `docs/Personas.md`(페르소나)는
   서로 합치지 않고 독립된 문서로 관리한다.
4. **사용자 결정 필요 시 질문**: 기술 스택, 정책, 범위 등 사용자의
   결정이 필요한 사항은 임의로 진행하지 않고 먼저 질문한다.
5. **의사결정 기록**: 기술 스택 선택, 정책 결정 등 주요 의사결정은
   `docs/DecisionLog.md`에 남긴다(무엇을, 왜 그렇게 결정했는지 포함).
6. **작업/오류 기록**: 진행한 작업, 발생한 오류와 그 수정 내역은
   `docs/WorkLog.md`에 기록한다.
7. **프롬프트 원문 저장**: 사용자가 준 주요 프롬프트 원문은
   `docs/PromptLog.md`에 그대로 저장한다.
8. **브랜치/PR 워크플로우**: 큰 기능이나 변경은 `main`에 직접 커밋하지
   않는다. 반드시 브랜치를 파서 작업하고 PR을 거쳐 병합한다.
   - "큰 기능/변경" 기준: 새 기능 구현, 여러 파일에 걸친 코드 변경,
     구조/스키마/설정 형식 변경, 마일스톤 단위 작업
   - 브랜치 네이밍: `feat/<기능명>`, `fix/<대상>`, `refactor/<대상>`, `docs/<대상>`
   - 작업 완료 후 PR 생성(`gh pr create`) — 본문에 변경 요약과 테스트/검증
     결과를 포함한다
   - 병합은 **squash merge**(`gh pr merge --squash`)로 한다 — main 히스토리는
     PR 단위 1커밋으로 유지
   - 병합 후 작업 브랜치는 삭제한다
   - 예외: **문서만 수정하는 변경**(코드 미포함)은 main에 직접 커밋/푸시 허용
9. **Git 작업에 AI 흔적 금지**: 커밋 메시지, PR 제목/본문, 브랜치명 등
   어떤 git 산출물에도 AI 사용 흔적을 남기지 않는다.
   - `Co-Authored-By: Claude ...` 트레일러 금지
   - `🤖 Generated with Claude Code` 같은 생성 도구 서명/링크 금지

# Required Artifacts (필수 산출물)

아래 문서들은 이 프로젝트의 필수 산출물이다. 각 문서는 존재해야 하며,
관련 작업을 진행할 때마다 최신 상태로 갱신한다. 누락된 문서를 발견하면
먼저 생성한다.

- `docs/ProjectContext.md` — 프로젝트 배경과 목표
- `docs/Plan.md` — 구현 전 작성하는 계획 문서
- `docs/IA.md` — 정보 구조(화면 목록, 화면 흐름)
- `docs/UserScenarios.md` — 사용자 시나리오
- `docs/Personas.md` — 사용자 페르소나
- `docs/DecisionLog.md` — 주요 의사결정과 근거
- `docs/WorkLog.md` — 작업 진행 내역, 오류와 수정 이력
- `docs/Process.md` — 작업 방식/프로세스(어떻게 일하는지에 대한 규칙과 절차)
- `docs/PromptLog.md` — 사용자 프롬프트 원문
- `docs/ReviewChecklist.md` — 완료 전 점검할 리뷰 체크리스트
- `docs/Research.md` — 기술/라이브러리 조사 및 리서치 내용
